Can AI Agents Generate Microservices? How Far are We?

Bassam Adnan1, Matteo Esposito2, Davide Taibi2,3, Karthik Vaidhyanathan1
1SERC, IIIT Hyderabad, India — 2University of Oulu, Finland — 3University of Southern Denmark, Denmark
IEEE International Conference on Software Architecture (ICSA 2026)
Paper arXiv Code Data

Abstract

Context. LLMs have advanced code generation, but their use for generating microservices with explicit dependencies and API contracts remains understudied.

Goal. We examine whether AI agents can generate functional microservices and how different forms of contextual information influence their performance.

Method. We assess 144 generated microservices across 3 agents, 4 projects, 2 prompting strategies, and 2 scenarios. Incremental generation operates within existing systems and is evaluated with unit tests. Clean state generation starts from requirements alone and is evaluated with integration tests. We analyze functional correctness, code quality, and efficiency.

Results. Minimal prompts outperformed detailed ones in incremental generation, with 50–76% unit test pass rates. Clean state generation produced higher integration test pass rates (81–98%), indicating strong API contract adherence. Generated code showed lower complexity than human baselines. Generation times varied widely across agents, averaging 6–16 minutes per service.

Conclusions. AI agents can produce microservices with maintainable code, yet inconsistent correctness and reliance on human oversight show that fully autonomous microservice generation is not yet achievable.

Study Overview

We evaluate 3 AI coding agents on their ability to generate microservices across 4 projects, 2 generation scenarios, and 2 prompt strategies—producing 144 implementations assessed for functional correctness, code quality, and efficiency.

Research Questions

RQ1: Functional Correctness & Code Quality

To what extent can AI agents generate microservices across different contextual scenarios, in terms of functional correctness and code quality?

  • RQ1.1 — Incremental generation: agents build within an existing system.
  • RQ1.2 — Clean state generation: agents work from requirements alone.

RQ2: Efficiency

How do AI agents compare in efficiency when generating microservices?

  • Token Consumption — Total input and output tokens.
  • Cost — Monetary costs based on each agent's pricing model.
  • Generation Time — Total time from start to completion.

Study Design

We selected 4 microservice systems (2 open-source, 2 private) ranging from 3 to 41 services, and 3 AI coding agents representing both proprietary and open-source models. For each project, we selected 3 microservices based on business logic importance, generating them under each experimental condition (4 conditions × 3 agents × 3 services × 4 projects = 144 total implementations).

Experimental Conditions

Each microservice is generated under two scenarios (how much existing code the agent can see) combined with two prompt strategies (how much guidance the agent receives), yielding 4 conditions per generation task:

Incremental Generation

The target microservice's code is removed, but the rest of the system (integration points, dependencies, other services) remains intact. Agents can explore existing architecture. Evaluated with unit tests.

Clean State Generation

All traces of the target microservice are removed—directory, config files, and API calls from other services. Agents must infer everything from requirements alone. Evaluated with integration tests.

P1: Minimal Prompt

Only the microservice name + path to requirements. Agents must explore or infer on their own.

P2: Prompt with Implementation Summary

Adds a generated summary covering functional scope, API responsibilities, integrations, and implementation notes.

Code Quality (RQ1)

We compared generated code against human-written baselines using SonarQube metrics: Lines of Code (LoC), Cyclomatic Complexity (CycC), and Cognitive Complexity (CogC). The top row compares by agent, the bottom row by experimental configuration.

Generated code generally exhibits lower complexity compared to baselines. Claude Code and Codex consistently produced code with 20–30% lower cyclomatic and cognitive complexity across all configurations, while maintaining comparable Lines of Code.

  • Claude Code and Code Qwen are faster (7–8 min avg) while Codex is slower (16.6 min avg) with outliers reaching 104 minutes

Key Findings

Summaries Can Hurt

Explicit implementation summaries (P2) degraded incremental generation. Agents anchored on high-level summary details, missing low-level specifics they would have discovered through codebase exploration with minimal prompts (P1).

Context Matters More Than Prompts

The choice of generation scenario (incremental vs. clean state) has a statistically significant impact on correctness (Z = 5.180, p < 0.001), while prompt type differences (P1 vs P2) did not reach significance (p = 0.140).

Data Contamination Effects

Agents performed substantially better on popular open-source repositories (PiggyMetrics, Train-Ticket) than private projects, likely reflecting training data memorization. This disparity persisted across all agents.

Concise ≠ Incomplete

Generated microservices are typically more concise than human baselines but achieve comparable test pass rates. Code verbosity does not correlate with quality—brevity should not be interpreted as insufficient.

Implications for Practice

  • Agent and prompt selection must be determined empirically rather than following general guidelines. Minimal prompts (P1) consistently outperformed detailed summaries for Codex and Claude Code, while Code Qwen required explicit guidance (P2) in clean state scenarios.
  • Plan for timeout risks and establish generation time limits. Codex showed concerning outliers reaching 104 minutes. Production systems should implement timeout mechanisms (e.g., 15–20 minute limits) and fallback strategies.
  • Expect performance degradation on proprietary codebases compared to benchmark results. Pilot evaluations on actual production microservices provide more realistic performance expectations than public benchmarks.
  • Anticipate evolution toward architecture-specific tooling. Current agents excel at implementation but lack architectural reasoning about organizational context and system-wide trade-offs.
